# lazycortex-specs — pending changes

**Status**: changed — `plugin.json` version `2.0.0` has not been published yet (last recorded release: `0.1.7`).

## Highlights since `0.1.7`

- `spec.install` Step 1 clarified — an `installed_plugins.json` entry scoped to a different project still counts as installed; only an absent or empty entry array aborts.
- `spec.request-handler` now ALWAYS marks at least one option as recommended in every operator-facing tick-list. Routing (multi-select): `★` on every candidate the specialist would tick itself, never silent. Each `[!question]` discrete-options sub-list: exactly one `★` for single-select framing ("Pick one", "or" between alternatives), one or more for multi-select ("Tick all that apply"). Zero recommendations is never the answer — if the specialist truly can't pick, the question is malformed and gets rewritten.
- Recommendation marker simplified to `★` only — the trailing `[recommended]` suffix is dropped from the agent's authoring contract and from the format spec. Routing rows and question options carry just the leading star; rationale prose follows the option text as usual.
- `spec.request-handler` agent body narrowed to its owned section. Content-clarification responsibility (scanning the user prose for ambiguities, writing `[!question]` callouts inline) is dropped from this expert — that work moves to a separate interpreter expert in the request-class main-writer chain. The handler now owns ONLY the `# Routing` section, writes the `#expert/spec-request-handler` ownership tag under the heading on every round, and never edits body prose or content-question callouts. The format spec example block now includes the tag.
- `test-interpreter` agent ships in `tests/specs/request/agents/` as the test-side main writer for the requests class — a minimal interpretation specialist that reads user prose, writes `[!question]` callouts (with `★` recommendation markers per the single/multi-select rules) adjacent to ambiguous statements, and folds operator answers into the body on subsequent rounds. Production consumers register their own interpreter expert; this test version covers the chain mechanics. Wired into the test worktree's `lazy.settings.json` so the requests class now dispatches `test-interpreter` (main writer) alongside `spec.request-handler` (routing writer) and the historian.
- `spec.install` Step 5c now emits the writer-group review-class schema (per the new `review.classes[].experts` object shape) — `routing` for `spec.request-handler`, `main` for the consumer-supplied interpreter, `history` for the historian. No `role:` keys; the old flat-array shape is rejected by the dispatcher with a `ValueError`.
- `spec.request-handler-protocol.md` and the request-format doc swept of `active_writer` / `final_check` vocabulary — the handler is described as the `routing` expert that owns `# Routing` and is paired with a consumer-supplied `main` interpreter; the agent body refers to "the main writer for the document" instead of "the active_writer".
- Routing intro callout switches from `[!question]` (operator-blocking — now reserved for content questions only) to `[!hint] Which targets apply? #review/hint` (section-internal advisory) so the routing surface never gates approval. Missing-invariant alerts and other in-review audit-trail callouts use the new `[!danger] ... #review/danger` tagged shape; clean audits use `[!summary] ... #review/summary`. The "class still unknown" placeholder inside Routing also uses `[!hint]` since the real clarification work belongs to the main-writer interpreter, not the routing writer.
- Adaptation to lazy-review v4 contract — the request-handler agent body, the `spec.request-handler-protocol.md` reference (now version 2), the `spec.request-protocol.md` reference, `spec.install` Step 5, `spec.request-attach`, `spec.set-stage`, and the per-file-stages reference are all swept clean of v1 vocabulary (`v_llm_sha` / `head_sha` / `file_content` / `v_llm_content` / `scaffold.strip` / `scaffold.reapply` / `V_human` / `Form D` / `lazy-core.runtime.routines.*`). Stale cross-links to `claude/lazycortex-review/rules/lazy-review.protocol.md` and `lazy-review.markup.md` now point at the canonical `references/lazy-review.doc-review-protocol.md` v2. The "foreign-owned callouts" concept is rephrased inline as "any callout whose first line lacks `#review/<x>` survives finalize" — no external rule lookup needed. Ownership-isolation is now described as dispatcher-enforced in code, not a self-policing rule on the agent.
- `spec.install` Step 5a and the `spec.request-handler-protocol.md` example block now seed the request-handler md-scan routine at `interval_sec: 5` (down from 30) — aligns with the 5-second baseline across the runtime.
- Same seed paths now declare explicit `timeout_sec: 60` for the request-handler md-scan routine — installers prescribe per-routine timeouts rather than leaning on a global daemon default.
- `spec.install` Step 5a and the `spec.request-handler-protocol.md` example also seed `priority: 20` for the md-scan routine — request dispatch is a fast scan + frontmatter-filter pass, should land before the slow `lazy-expert.pump` (priority 100) in each daemon tick. The md-scan also gets a `review_active: [null, false]` filter clause so it stops re-dispatching once a request is in review — kills the no-op-commit spam previously visible in git log.
- `spec.request-handler` agent body purged of lazy-review implementation leak: removed the literal protocol filename + `§ Lifecycle / Naked → review-open` paragraph reference (was making Sonnet scan disk via `find /Users/...` looking for the file when the md-scan routine forgot to declare `protocol:`); dropped the `claude/lazycortex-review/bin/body.py` path, `OwnershipViolation` class name, and `body.strip_for_main_writer` / `body.strip_for_section_writer` function names from the ownership-tag invariant; dropped the "tags-mirror parity is part of the protocol" and "the contract the apply phase reads" lifecycle-internal explanations; "Operator's tick is still the only signal that lands the apply phase" → "the only signal that counts". Rules unchanged, justifications no longer leak machinery details.
- **Breaking — `spec.install` Step 5a md-scan routine now uses `expert:` + `request:` instead of the legacy `agent:` field**. Matches the lazycortex-core routine-types schema unification (every type accepts EITHER `command` OR `expert + request`). Consumers must re-run `spec.install` (or hand-edit `lazy.settings.json[routines][spec.request-handler]`: rename `agent: spec.request-handler` to `expert: spec.request-handler` and add `request: {file: "{file}"}`).
- **Apply transition rewritten as deterministic Python primitive** (`bin/apply_request.py`, replacing the prior LLM `spec.request-apply` agent). The post-finalize routine now dispatches via `command: [lazycortex-specs, apply-request]` — no LLM hop, no nested-skill MANDATORY-preamble hijack, no echo-and-exit failure mode. The worker parses the resolved `# Routing` prose (class verdict + spawn / attach targets), calls the `scaffold-asset` CLI for spawns, distributes the request body Tier 3 fallback into each entity's WTR doc, maintains `spec_source_requests` frontmatter + the `## Requests` projection inside `# Sources`, appends the folder-note `## Source requests` bullet, opens a review cycle on every populated doc via `lazycortex-review start`, and stamps terminal markers on the request file in one atomic commit. Tier 1.5 LLM-judgment body distribution is dropped from the automated path (the operator-invoked `/spec.request-attach` skill still keeps it for manual cases). The `spec.request-apply` agent file is removed; the expert registration is no longer needed in `lazy.settings.json[experts]`. `spec.install` Step 6b switches the routine shape from `expert:` to `command:` and adds review classes for spawned `design.md` (main = designer) and `plan.md` (main = planner) paths so the daemon dispatches the right writer for each freshly-spawned spec doc — without these classes, spawned docs sat at `[!hint] Waiting #review/in-process` forever.
- `scaffold-asset` CLI primitive added (`bin/scaffold_asset.py`) — deterministic Python that scaffolds a full entity folder (folder-note + per-category authored docs + per-doc default `spec_source_docs` projection + history line) under the product's `spec_path`. Replaces the prior `spec.request-spawn` → nested-skills chain for the apply path; the `spec.create-asset` wizard delegates Step 5 (mechanical write) to the same primitive while keeping its operator-facing question flow.
- `spec_source_docs` body bullets render with shape-aware default displays — product-level `<spec_path>/docs/<role>` references render as `<product> — product <role>`, sibling-asset references render as `<slug> — <role>`, so the rendered `## Docs` sub-section reads sensibly without operator rewrites. Frontmatter stays a target-only canonical list; the writer preserves operator-edited displays across re-projections by matching on the wikilink path.
- `request_status` enum collapsed from `{draft, accepted, rejected, spawned}` to `{draft, accepted, rejected}` — the `spawned` value was redundant once the success callout body lists every resolved-target wikilink (whether attached or spawned). Mirror tag values track the new enum; `spec.doctor` and `spec.request-attach` updated.
- `spec.request-apply` execution contract consolidated into agent body (and now into the Python worker's module docstring + `_parse_routing` function) — the prior standalone `references/spec.request-apply-protocol.md` is removed. The single mechanical-purpose worker carries its input shape, completion sequence, terminal markers, and side-effect bounds inline.
- `spec.request-spawn` skill narrowed to a thin three-step nested wrapper around the `scaffold-asset` CLI + delegation to `spec.request-attach`. Declares `execution-discipline-waiver` per the new `lazy-core.skill-writing § 1.5` rule — nested skills invoked from another agent's body do not carry the canonical Execution-discipline preamble (it would hijack the outer agent's mental model and trigger sub-skill-return-as-deliverable drift).
- Apply worker corner-case sweep: display gloss in folder-note `## Source requests` and spawned-doc `## Requests` projections now reads the request's actual H1 title (not `# Routing`, which used to win when read before strip); `spec.install` Step 6f added — when 6e wires the design.md / plan.md classes for spawned spec docs, the consumer MUST also extend `routines[lazy-review.scan].paths` with the matching nested globs (md-scan uses `PurePath.match` where `*` does NOT cross `/`, so per-level globs are required; without the extension the daemon never scans spawned docs and the new classes are dead).
- Apply ↔ router contract switched to a structured `<!-- routing-decision -->` block — line-oriented, one decision per line (`spawn <kind> <slug>` or `attach <repo-relative-folder-note-path>`). The router agent's body now mandates the block at the end of every settled `# Routing` section after the operator-facing prose; the Python apply worker reads ONLY the block when present and ignores the surrounding prose entirely. This closes the class of bugs where each LLM router run invented a new prose shape (`Spawn ... entity — \`slug\``, `kind: slug`, `spawn \`<full-path>\``, `spawn · product X · kind Y` without slug, …) and the apply parser had to catch up with another regex. New apply contract: parse tokens, not natural language. Legacy prose-regex fallback retained for migration so files finalized before the switchover still apply.
- `spec.install` adopts the quiet-install model — install scope derived from `installed_plugins.json`, rule/template drift resolved by the 3-case file-sync policy (write/merge silently, prompt only on a genuine conflict, orphans kept), the request-handler md-scan routine and spawned-doc review classes gated on the project `daemon.enabled` flag, and the repo default-language + first-product steps kept as read-first questions.
- Gate cascade hardened. `spec.gate-tick` promotes `spec_stage` draft→approved when a gate is fully ticked and atomically commits the readiness-callout drop/withdraw in the same step; `spec.flip-gate` auto-promotes an empty `plan.md` to `draft` as the design gate flips and atomically commits its folder-note edit. Each transition lands as one commit instead of leaving a dirty tree for the next pass to trip over. A four-bug cascade sweep also fixed the source-request display gloss, the `lazy-review.scan` per-level path globs for spawned docs, a `MAX_ITERATIONS` guard, and the `[!question]`-callout invariant.
