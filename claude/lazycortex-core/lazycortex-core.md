# lazycortex-core — pending changes

**Status**: changed — `plugin.json` version `3.2.0` has not been published yet (last recorded release: `3.1.1`).

## Highlights since `3.1.1`

- `default-tiers.json` seeds three new dispatcher rows for the forthcoming `lazycortex-experts` plugin (`lazy-experts.interpreter` and `lazy-experts.planner` default `sonnet`; `lazy-experts.designer` defaults `opus` — heaviest reasoning of the three). Existing vaults pick them up by re-running `/lazy-core.agent-models`.
- Expert runtime contract § Working tree strengthened — when an expert invokes a sub-skill that modifies tracked files (`lazy-review.start`, `spec.set-stage`, similar helpers), the sub-skill's writes count as the expert's writes and MUST be committed before exit. Preferred fix: sub-skill commits itself atomically; fallback: caller wraps with a final `git add -A && git commit`. `git status --porcelain` MUST be empty at expert exit — uncommitted dirt still halts the daemon.
- Expert runtime job_id stabilised via `realpath` resolution; dead-marked job dirs are now parked on redispatch instead of overwritten — recovers from interim refactor regressions and gives the daemon a clean restart path.
- Daemon supervisor gains an opt-in `--dev-mode` flag that makes `lazy.runtime.sh` prefer plugin sources under `<repo-root>/claude/<plugin>/` over the cache — useful when the host repo is itself a plugin-authoring vault. `/lazy-core.install` asks once and persists the choice at `lazy-core.runtime.supervisor.dev_mode`; existing consumers pick up the new shim on re-install via a content-diff refresh.
- `lazy-core.expert-protocols-contract.md` § 4.2 example vocabulary updated — the `role` strings table now lists `main` / `<section>` / `final` / `auto` / `repair`, matching the new lazy-review writer-group schema instead of the legacy `active_writer` / `final_check` doublet.
- Runtime daemon remote sync no longer swallows git errors and continues in a broken state. Pre-tick now fetches, compares HEAD against `origin/<branch>` via merge-base, and halts on diverged history (`reason: git_pull_diverged`) instead of risking a silent overwrite of operator commits. Post-tick replaces the bare push with a fetch + rebase + push retry loop (`POST_TICK_MAX_PUSH_ATTEMPTS = 3`); rebase conflicts cleanly discard the tick's work and let the next tick redo it on the operator's base, while exhausted retries halt with `git_push_failed`. Any other unexpected git failure halts with `git_remote_unavailable`. `/lazy-runtime.recover` gains a `manual-fix` mode that prints reason-specific repair guidance and clears the halt block once the operator has resolved the underlying state by hand.
