# lazycortex-core — pending changes

**Status**: changed — `plugin.json` version `4.0.0` has not been published yet (last recorded release: `3.2.0`).

## Highlights since `3.2.0`

- `expert_runtime.dispatch_job` is now an atomic job-bundle primitive: one call writes the entire `<jobs>/<expert>/<job_id>/` tree in pump-safe order (config.json + source/context/result + request.json, then `READY` touched LAST) and looks up `git_author` / `aspects` / `arguments` from `settings.experts[<expert>]` itself. Callers no longer compose around it; the previous spawn-race (pump picked up READY before sibling files existed) is architecturally impossible.
- New `lazycortex-core dispatch-job` and `lazycortex-core collect-job` CLI subcommands publish the queue surface for sibling plugins. JSON-in via stdin, JSON-out via stdout; resolved via `$LAZYCORTEX_PLUGIN_DIRS` env. Inter-plugin invocations always cross this CLI boundary — direct Python imports between plugins are forbidden.
- Internal helpers `routine_types._write_job_config` and `_resolve_expert` removed — the work they did is now owned by `expert_runtime.dispatch_job`. The four routine-type handlers (`dispatch_md_scan` / `dispatch_inbox` / `dispatch_schedule` / `dispatch_git`) collapse to a single `dispatch_job(...)` call each, passing only routine-side fields (protocols, source/result/context).
- `expert_pump` no longer guards on "config.json: both protocols and aspects empty". Specific-domain agents (test-designer, lazy-review.historian, doc_doctor, …) are self-contained — their `.md` file IS the full instruction set; the composition path (generic agent + aspects + protocol) is one valid spawn shape, not the only one.
- Daemon now defaults missing `routines[<name>].interval_sec` to 5 seconds (was a `KeyError`). Documented examples in `lazy-core.runtime-schema.md` + `help/walkthroughs/setup-routine.md` and the `DEFAULT_EXPERT_PUMP` bootstrap entry follow the same 5s baseline.
- Daemon and expert-pump hardcoded ceilings are now surfaced as settings keys with the same defaults: `daemon.default_routine_timeout_sec` (was `DEFAULT_TIMEOUT_SEC = 300`), `daemon.expert_pump.claude_spawn_timeout_sec` (was `timeout=900` literal in `_process_one`), and `daemon.expert_pump.retry_delays_sec` (was `RETRY_DELAYS = [1, 2, 4, 8, 16]`). Lookup chain per routine: routine `timeout_sec` > `daemon.default_routine_timeout_sec` > 300. Schema documented in `references/lazy-core.runtime-schema.md`.
- **Breaking** — pump now processes AT MOST ONE Claude spawn per invocation, then returns; subsequent READY jobs land on subsequent pump ticks. The in-loop retry over `[1, 2, 4, 8, 16]` backoff delays is gone — each Claude attempt is now a single try at routine-timeout granularity. The two settings keys that exposed those retries (`daemon.expert_pump.claude_spawn_timeout_sec` and `daemon.expert_pump.retry_delays_sec`) are removed; `daemon.default_routine_timeout_sec` is removed too — only per-routine `routines[<name>].timeout_sec` overrides matter now (code default 300s remains as fallback). The pump's `subprocess.run(claude, ...)` no longer carries an inner `timeout=` — the routine-level ceiling (default 900s in `DEFAULT_EXPERT_PUMP`) is the only Claude-spawn ceiling.
- **Breaking** — pump main loop now skips jdirs carrying a `DEAD` marker. Previously pump tried again on every tick regardless of DEAD, which made `_detect_dead_jobs` cosmetic. With this change DEAD is a real terminal-pending state until recovery clears it.
- New `lazy-recover.dead-jobs` routine (hourly, registered by `bootstrap_default_routines` alongside `lazy-expert.pump`) scans `.experts/.jobs/<expert>/<job>/` for DEAD-marked jobs and either retries (clears DEAD + dead.json + PID + transcript) or writes `diagnosis.json` for human triage. Retry budget defaults to 3 attempts (overridable via `LAZY_RECOVERY_MAX_ATTEMPTS` env). `_process_one` now bumps an `attempts` counter file before each Claude spawn so the recovery routine has persistent attempt history across pump kills.
- New CLI subcommand `lazycortex-core recover-dead-jobs` exposes the recovery routine for direct operator use; implementation lives in `bin/dead_recovery.py`.
- **Fix** — when the daemon is halted (`state.daemon_halted` block present, set by a routine that left the working tree dirty or by a git failure), the outer `run()` loop now sleeps `polling_interval_sec` directly instead of routing through `compute_sleep`. Previously the halted-state silent-return in `_run_iteration` left `last_run` untouched, `time_until_next_due` returned a large negative, and `compute_sleep` produced `time.sleep(0)` → 100% CPU busy-loop with no progress and no log entries.
- Dirty-tree halts now auto-clear once the tree becomes clean. On each iteration, before the halt early-return, the daemon checks: if `daemon_halted.reason == "uncommitted_changes"` AND the working tree is now clean, it pops the halt block, logs an `_auto_recover` line, and proceeds to dispatch routines. Other halt reasons (`git_pull_diverged` / `git_push_failed` / `git_remote_unavailable`) still require manual `/lazy-runtime.recover` — those need human investigation, not just a clean tree.
- `DEFAULT_EXPERT_PUMP.timeout_sec` bumped 900 → 1800 (30 min). Long-running specialists (historian / interpreter on rich documents) routinely take ~10 min per spawn and started hitting the 15-min ceiling. Raise further only when consistently hitting timeout in normal operation.
- **Breaking** — replaced the dumb counter-based `lazy-recover.dead-jobs` routine with autonomous `lazy-runtime.doctor` agent. The new doctor is a Claude expert that reads dead-job context (request.json, dead.json, error.json, transcript tail, attempts counter) and dirty-tree halt state, then decides per case: retry (clear DEAD), revert-and-retry (`git checkout` Claude's incomplete edits then clear DEAD), permanent-fail (write diagnosis.json for human triage), or commit-system-noise (commit `.DS_Store` / `.obsidian/workspace.json` etc. so the daemon resumes). Decisions are reasoning-driven, not regex-driven. Deleted: `bin/dead_recovery.py`, CLI subcommand `recover-dead-jobs`, `DEFAULT_DEAD_RECOVERY` routine. Added: `bin/doctor_tick.py` (hourly trigger), CLI subcommand `doctor-tick`, `agents/lazy-runtime.doctor.md` (the agent), `DEFAULT_DOCTOR_TICK` routine. Consumers must register the `lazy-runtime.doctor` expert in `settings.experts` and replace the old routine entry in `settings.routines`.
- `recover.py` extended with three primitives the doctor agent uses via Bash: `revert_files(repo, paths)` (`git checkout HEAD -- <paths>`), `clear_dead_job(jdir)` (unlink DEAD/dead.json/PID/transcript/error/partial-response.json, preserve attempts counter + READY), and `permanent_fail(jdir, diagnosis)` (write diagnosis.json, keep DEAD). The existing operator-facing `/lazy-runtime.recover` skill continues using the old `read_halt` / `cleanup` / `resume` / `is_clean` primitives unchanged — same module, two consumers (operator wizard vs autonomous doctor).
- `experts.<name>.model` setting added — pump reads it from `config.json` (snapshot by `dispatch_job`) and passes `--model <m>` to the `claude` subprocess. Without the setting, Claude inherits whatever the global Claude Code config picks (today: Opus 4.7 for most installs). Setting `model: "sonnet"` per expert cuts both latency (~3-5× faster than Opus) and output tokens (~5× cheaper) for tasks that don't need Opus-level reasoning — historian's one-sentence narrative, request-handler's triage with rubric, doctor's runtime diagnosis. Bump to Opus per-expert when a specific specialist needs it.
- Two new routine fields — `priority` (int, asc; default 100) and `ignore_halt` (bool; default false). `due_routines` sorts due routines by priority each tick, so fast mechanical routines (review-tick banner repaints, md-scan dispatchers) finish across all files BEFORE the slow `lazy-expert.pump` (priority 100) spawns the next Claude. `ignore_halt` lets a recovery routine run even when the daemon is halted or the working tree is dirty — `lazy-runtime.doctor` uses this so it can triage the very state that's blocking everyone else. `_run_iteration` no longer early-returns on halt; it passes a `system_stuck` flag into `due_routines` which filters non-`ignore_halt` routines. Post-routine dirty-tree halt-check also skips when the routine has `ignore_halt: true` — doctor may intentionally leave dirt (a written `diagnosis.json`) without triggering a halt of its own.
- Journal `.logs/lazy-core/runtime/<date>.jsonl` now skips no-op routine ticks. The heuristic in `_is_no_op_log`: short successful tick (exit 0, dur ≤ 1.5s, no error) whose stdout signals no work happened — pump with `processed=0 cleaned=0`, md-scan / inbox / schedule with `dispatched_count: 0`, or `lazy-review.tick` whose actions are all `kind: skip`. Metrics still record the tick (latency, queue depth, error rate stay accurate). Cuts journal growth from ~12k lines/day to ~hundreds — Obsidian-friendly without losing audit trail.
- Universal expert-runtime contract (`references/lazy-core.expert-runtime-contract.md`) gains generic `## Protocol awareness` and `## Aspect awareness` sections — the boilerplate that used to be duplicated inline in three lazy-experts agent bodies (interpreter / designer / planner) now lives once in the contract that every expert spawn loads via `--append-system-prompt-file`. Persona-only agent bodies; the mechanism is system-shared.
- `doctor_tick.py` gains a `_all_known_protocols` helper that collects every routine-declared `protocol:` / `protocols:` from `settings.routines` and passes the full list to the doctor when dispatching. When the doctor is triggered (DEAD job or stuck halt), it sees in its prompt every contract that any expert in the system was meant to fulfill — so it can read the relevant one to reason about a dead-job's expected I/O without zashivat' that knowledge into the doctor's body. Trigger gate unchanged: doctor still only dispatches when there's actual work (no DEAD + no halt → `triggered: false`, no Claude spawn).
- **Fix** — `routine_types.validate_routine_entry` now accepts the `priority` and `ignore_halt` fields in `COMMON_ALLOWED`. The fields were introduced project-wide by the priority+ignore_halt feature but never added to the validator's allowlist; `bootstrap_default_routines` writing `priority: 100` / `priority: 30, ignore_halt: True` for the built-in `lazy-expert.pump` and `lazy-runtime.doctor` routines raised `RoutineConfigError: unknown field(s)` on every fresh-install attempt. Existing daemons survived because the validator runs on register, not on read — fresh-install was the only path that bit. Caught by the spec test `test_expert_runtime_spec.py::TestBootstrapDefaultRoutines`.
- **Sandbox expert spawns by default**. Three-part fix: (1) `runner` now calls `install_sandbox_settings(repo)` after `set_plugin_dirs`, generating `<repo>/.claude/settings.json` with `sandbox.filesystem.allowRead/allowWrite` whitelisting only the worktree + each declared `--plugin-dir` path, plus `permissions.deny: ["Bash(find /*)", "Bash(grep -r /*)", ...]`. Idempotent — preserves any operator-authored keys, only rewrites the three blocks it owns. (2) `expert_pump` switches `claude -p --permission-mode bypassPermissions` → `--permission-mode dontAsk` so deny rules actually fire (bypassPermissions skipped them). (3) `expert_pump` also propagates `LAZYCORTEX_PLUGIN_DIRS` env (set by `runner`) into `claude -p --plugin-dir <path>` flags per entry, so the spawn's Skill discovery sees every sibling plugin natively — without this the agent fell back to `find /Users/...` scans through the dev-vault when it couldn't resolve a sibling-plugin skill name (`lazy-review.start`, `spec.request-classify`, etc.). Net effect: expert spawns are now sandboxed to worktree+plugin-dirs at the OS level (macOS Seatbelt); any `Bash(find /...)` fails immediately so a misguided agent surfaces an error instead of burning minutes on Dropbox-checkout scans.
- **Breaking — unified routine-types schema**. Every routine type (`subprocess`, `inbox`, `schedule`, `git`, `md-scan`) now accepts EITHER `command` (subprocess) OR `expert` + `request` (job-queue dispatch), enforced uniformly via `_validate_command_or_expert`. The `md-scan` type's old `agent:` field is gone — consumers migrate to `expert:` + `request:`. `dispatch_md_scan` grows a PID-based dedup helper (`.experts/.subprocess-locks/<routine>/<sha>.json`) so `command:`-shape md-scan routines spawn non-blocking with the same one-active-per-key guarantee that `dispatch_job` provides for the `expert:`-shape. `dispatch_subprocess` / `dispatch_inbox` / `dispatch_git` also pick up the alternate shape — schedule was the pattern's early adopter; every type now matches it. New regression suite `tests/core/runtime/test_routine_types_unified.py` pins the EITHER/OR invariant across all 5 types.
- **New `consume-job` lifecycle primitive**. `expert_runtime.consume_job(repo, expert, job_id)` writes a `CONSUMED` marker inside the job dir, and the new `lazycortex-core consume-job` CLI subcommand exposes it to sibling plugins (JSON `{expert, job_id}` on stdin, idempotent, returns `{status: ok}`). `dispatch_job`'s active-check now also skips jobs with `CONSUMED` for symmetry. Purpose: close the lifecycle gap between `DONE` (producer-side, written by `expert_pump` when the agent finishes) and the consumer's "I've applied or discarded this response". Without `CONSUMED`, a consumer that looks up its expert's DONE job by `dedup_key` keeps re-reading the same stale response forever — the operator-conflict snapshot check inside the consumer discards it, but no fresh dispatch is queued, and the pipeline locks up. With `CONSUMED`, the consumer marks the job retired after applying or discarding, the next lookup reports `missing`, and `dispatch_job` queues a fresh attempt on the next tick. Forensic window unchanged — the job dir stays on disk until the standard cleanup TTL in `expert_pump._maybe_cleanup` removes it. First consumer is `lazycortex-review`'s dispatcher; the primitive is generic so any future component that uses the lookup-by-key pattern inherits a clean lifecycle by default.
