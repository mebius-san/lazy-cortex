# lazycortex-python — pending changes

**Status**: changed — `plugin.json` version `1.4.1` has not been published yet (last recorded release: `—`).

## Highlights since `—`

- First ship-ready cut of the Python coding-discipline plugin. Distributes one shared spec — three short path-scoped rules in `.claude/rules/` (style, docstrings, tests), five reference guidelines in `${CLAUDE_PLUGIN_ROOT}/references/`, a canonical `python-template.py` scaffold — across any consumer repo via `/lazy-python.install`. The plugin is the single source of truth for "how this org writes Python"; consumers don't carry the canon themselves, just the overlay deltas under `docs/guidelines/`.
- Checker stack ships ready-to-run. Three shipped binaries (`pcf` for fast format / import / docstring-shape critical-fails, `toi` for test-of-intent gates, `pch` for PyCharm-inspect static analysis) plus two zsh aggregators (`chk` for full style/type sweep including mypy + pylint, `tst` for pytest) live under `${CLAUDE_PLUGIN_ROOT}/bin/`. Per-consumer thin wrappers `chk-py` and `tst-py` land in `<repo>/cli/` (per the repo-root CLI convention) so the operator can run `./cli/chk-py all src/foo.py` from any terminal session, and the `-py` suffix is reserved — sibling language plugins will own their own (`chk-cpp`, `tst-go`, etc.).
- PostToolUse hook runs the cheapest checker (`pcf`) automatically after every Edit/Write on a `.py` file and returns violations to Claude as `additionalContext` — feedback arrives in the next turn without an explicit skill invocation. First line of defence; the `/lazy-python.check-style` skill (six-step deep manual review + full `chk-py all` run) is the second; the operator-driven `chk-py all` from the terminal is the third.
- Two paired agents do the actual writing. `lazy-python.docstring-writer` generates module/class/method docstrings against the canonical guidelines (no implementation leakage, Obsidian-compatible LaTeX, descriptive-only DOC comments). `lazy-python.test-writer` generates pytest-style tests with the file's intended-failure annotations (`DO NOT RELAX`, `INSTANT FAIL`) preserved as safety nets.
- Eight-phase idempotent install (`/lazy-python.install`): mirrors plugin rules into `<repo>/.claude/rules/`, deploys CLI wrappers into `<repo>/cli/`, bootstraps `pyproject.toml` with `[tool.pcf]` / `[tool.toi]` / `[tool.pch]` (mandatory) plus `[tool.pytest.ini_options]` / `[tool.mypy]` / `[tool.pylint]` (recommended), probes for the optional PyCharm `inspect.sh`, scaffolds empty project-overlay stubs under `<repo>/docs/guidelines/`, registers `python-template.py` in the consumer's scaffold registry, optionally appends a discipline pointer to `<repo>/CLAUDE.md`, and registers the PostToolUse check-style hook.
- Eleven-check read-only audit (`/lazy-python.audit`) verifies every install artifact: rules byte-mirror the plugin canon, references cited from rules resolve, shipped artifacts are all present and executable, CLI wrappers are deployed under `<repo>/cli/` with substituted bin paths, mandatory pyproject sections are present, the inspect.sh probe still resolves, overlay stubs carry the canonical header, the scaffold registry entry points at the consumer-local `.claude/templates/python/python-template.py`, `CLAUDE.md` carries the discipline bullet, the PostToolUse hook is registered, and the venv probe-then-fallback chain mirrors `_ensure_venv.sh`. 25 unit tests across `tests/python/audit/` pin each check's PASS / WARN / FAIL contract.

Idempotence is enforced — re-running install on a clean tree is a no-op.
- Venv resolution is probe-then-fallback. `_ensure_venv.sh` looks at `$VIRTUAL_ENV` first, then `<repo>/.venv`, then `[tool.lazy-python].venv_path` if declared, then `${CLAUDE_PLUGIN_DATA}/venv/` as a uv-managed fallback. The hook never blocks on venv setup; consumer projects without any of the above get a managed fallback at first run.
