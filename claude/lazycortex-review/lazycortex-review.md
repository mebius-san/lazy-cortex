# lazycortex-review — pending changes

**Status**: changed — `plugin.json` version `5.0.0` has not been published yet (last recorded release: `0.1.0`).

## Highlights since `0.1.0`

- Full bin/ rewrite against a new functional spec. The bin tree now splits cleanly by responsibility: primitives (`errors`, `frontmatter`, `parser`, `git_ops`, `job_ids`), document parts (`body`, `banner`, `history`, `edit_markup`), assembly (`payload`, `reapply`), orchestration (`state_machine`, `dispatcher`), and the slash-command verbs (`start`, `stop`, `finalize`, `status`) plus the `lazy_review` CLI.
- Ownership isolation is now enforced in code, not in agent prompts. `reapply.py` restores the operator's body byte-for-byte when a section writer strays outside its owned H1; `payload.validate_frontmatter_overlay` drops any agent attempt to write the reserved keys `review_active` / `review_round` / `approved` before they reach the doc.
- Banner-tick invariant: every change to the top status banner is its own mechanical commit, never bundled with an agent's content commit. The state machine's `banner-repaint` branch short-circuits all other branches — even when `missing_history_entry` or `main_chain_pending` would also fire, the banner repaint wins and the rest waits one tick. The operator sees "whose turn it is now" within ~5 seconds of any state change.
- Frontmatter edits are surgical line-edits, never `parse → render`. Block-style YAML values (`tags:\n  - foo`), inline arrays, comments, and quoting style all survive byte-for-byte through `set_field` / `unset_field`. Regression-tested.
- Doc-review protocol re-published at v2 (`references/lazy-review.doc-review-protocol.md`). Trimmed from 517 lines to ~190: the wire shape, the per-kind outcome table, the closed callout vocabulary, and the four edit-annotation styles — nothing more. Per-role mechanics that the dispatcher now enforces in code are gone from the protocol doc. Includes an explicit Migration-from-v1 section.
- `review.classes[].experts` shape change: from a flat array with `role:` keys to a dict keyed by writer-group (`main`, `<section>`, `history`, `final`). The dispatcher reads writer groups; the historian default falls back to a built-in when `history` is omitted; per-section groups encode their H1 title (`routing` → `# Routing`, `open_design_questions` → `# Open design questions`).
- 163 unit tests cover every primitive, the banner state machine, the body strip/reassemble contract under all three roles, the ownership-isolation refusals, the per-priority state-machine branches, the slash-command verbs, and the banner-first-tick integration scenario.
- Slash commands as thin dispatchers: `lazy-review.start` / `stop` / `finalize` / `status` / `iterate` are now single subprocess calls over `bin/` scripts with atomic commit-on-exit and clean working tree on return. The wrapper `lazycortex-review` CLI exposes `tick` / `iterate` / `status` / `start` / `stop` / `finalize` subcommands and writes a per-action JSONL run log under `.logs/lazy-review/runs/<UTC>.jsonl`.
- Removed the standalone webhook server and the in-plugin `lazy-review watch` polling mode. Remote work (`git pull`/`push`/`rebase`/`reset`) is `lazycortex-core`'s runtime daemon's responsibility, not lazy-review's; the plugin's git_ops surface is intentionally narrow (no `push` / `pull` / `fetch` / `rebase` / `reset_hard` exports) and a unit test pins that invariant.
- v3 rules folder cleared: both `rules/lazy-review.markup.md` and `rules/lazy-review.protocol.md` deleted. Their functions are covered without auto-load — verb discovery happens through the skill-listing the harness already shows at session start; the dispatch contract lives in `references/lazy-review.doc-review-protocol.md` v2 for on-demand reading; the cheatsheet for new consumers is `/lazy-review.help`. Loading either rule on every markdown session was a tax on context for a sub-case rarely exercised.
- Dispatcher now writes the per-job `config.json` for every job it queues (main / section / final writer, historian, doc_doctor), populated with the doc-review protocol id and the expert's git author. Previously the dir was created with only `request.json` + `READY` + `source/` / `result/`, and the expert pump rejected every job with `config.json missing` — the v4 chain could never spawn an agent end-to-end. The new order writes config.json + source/result BEFORE `dispatch_job` touches READY, so the pump's first read always finds a complete dir (no spawn-race).
- Cross-plugin coupling dropped: the dispatcher no longer reaches into `lazycortex-core`'s `bin/` via a `sys.path` filesystem walk. All job dispatch + collection now flows through `lazycortex-core`'s public CLI subcommands (`dispatch-job` / `collect-job`) resolved via `$LAZYCORTEX_PLUGIN_DIRS`; the previous job-dir composition (mkdir + write source/result/config.json + READY) lives entirely in core. See the inter-plugin boundary contract for the pattern the dispatcher now uses.
- `install.py` now seeds `lazy-review.tick` at `interval_sec: 5` (down from 60) — matches the 5-second baseline the rest of the runtime ships with.
- `install.py` also seeds explicit `timeout_sec: 60` for `lazy-review.tick` — installers prescribe per-routine timeouts rather than leaning on a global daemon default.
- `install.py` seeds `priority: 10` for `lazy-review.tick` — banner-repaint + dispatch passes are mechanical and should finish across all files BEFORE the daemon yields to `lazy-expert.pump` (priority 100) for the next slow Claude spawn. Operator sees status callouts update on every file within seconds of any state change.
- `lazy-review.historian` agent frontmatter narrowed: `tools: Read` only (was `Read, Glob, Grep`). The agent body now carries an explicit constraint: read only the two staged files — `source/<file>` (current HEAD, pre-stripped of review-protocol metadata) and `context/prior-<file>` (HEAD~1, same stripping). Removing Glob/Grep eliminates the historian's incentive to scan `.logs/` or other docs to "understand context" — burning thousands of tokens on noise. The routine already pre-filtered the diff to exactly what historian needs to narrate.
- `lazy-review.historian` agent body sweeps off doc-review-protocol-specific vocabulary — the inline dump of stripped metadata structure (`top banner, # History section, approve checkbox, system callouts tagged #review/<x>, frontmatter keys review_active / review_round / approved`) becomes a single neutral "bookkeeping metadata" reference; "Tags are recognition markers for the protocol's gating predicates" becomes "for downstream automation". Behavior unchanged — agent body just no longer leaks protocol internals.
- `bin/install.py` now seeds `protocol: lazycortex-review:lazy-review.doc-review-protocol` on the `lazy-review.tick` routine. Pairs with the lazycortex-core doctor change that collects every routine's declared protocol — without this seed, fresh consumer installs let the doctor triage a dead historian / final-writer job without ever seeing the contract that job was meant to fulfill.
- **Fix** — `bin/dispatcher.py` `_core_dispatch_job` now auto-injects `protocols: ["lazycortex-review:lazy-review.doc-review-protocol"]` into every dispatch bundle that doesn't already carry one. Every expert this dispatcher fires (review main / section / final writer, historian, doc_doctor) runs under the doc-review protocol by design — but the three call sites were passing bundles without `protocols`, so core's `dispatch_job` wrote `protocols: []` into config.json, and the spawn's user-message prompt carried no `- protocol:` line. Sonnet then either guessed at the response shape (causing `response_invalid: outcome='edited' on kind='review' requires non-empty 'history_entry'` errors) or chased the protocol filename via `find`. Single point-of-injection in the helper covers all three call sites and any future ones.
- **Breaking — `lazy-review.tick` subprocess routine retired in favour of `lazy-review.scan` md-scan routine**. The runtime daemon's per-document driver is now `command: ["lazycortex-review", "process-file"]` on an `md-scan` routine that globs each class's `paths:`, filters by `review_active: [null, true]`, and spawns one `process-file <abs-path>` subprocess per matching file with PID-based dedup (one active worker per file at a time). The legacy `tick` CLI subcommand is retained for the operator-facing `/lazy-review.iterate` one-shot scan; it is no longer invoked from the routine. `bin/dispatcher.py` extracts `process_one_file(repo, file_path)` from `run_tick(repo)` — `run_tick` is now a thin loop around the per-file driver. `lazy-review.configure` skill now also syncs the routine's `paths:` glob list from `review.classes[].paths` on each Phase 5 write so the daemon scans the same set the dispatcher routes against.
- **Fix — completed jobs no longer block re-dispatch on the same key**. `apply_action` now marks the job via `lazycortex-core consume-job` (touches a `CONSUMED` marker inside the job dir) after applying or discarding the response. `_find_done_job` skips jobs with `CONSUMED` (treats as missing), so the next tick dispatches fresh instead of re-reading the same stale response forever. Pre-fix symptom: a daemon restart on a worktree with prior DONE jobs left the historian / main writer silently re-using yesterday's answer (or stuck on `discarded:operator-conflict`). The lifecycle gap was systemic — `DONE` (producer-side, written by `expert_pump`) and "consumer is finished" were conflated, so any consumer using the lookup-by-key pattern would have hit the same bug. The new `CONSUMED` marker is the consumer-side signal; the forensic window (job dir on disk for the standard cleanup TTL) is preserved.
- **Fix — approve-mirror mechanical step precedes banner-repaint**. The operator's approve gesture is a `- [x] approve the whole document` tick *inside* the Ready banner body. The next banner-repaint would otherwise rewrite that callout (operator commit transitions banner Ready → in-process), eating the approve checkbox without copying it anywhere — finalize-chain would never fire because frontmatter `approved` stayed `false`. New state-machine priority slot `approve-mirror` (priority 3, ahead of banner-repaint) writes `approved: true` into frontmatter via a mechanical commit before the next banner-tick repaints. By the time the Ready callout is stripped, the operator's intent is durable as metadata; finalize chain takes over on the following tick.
- **Breaking — section schema redesign**. `review.classes[].experts.{validation,terminal}` теперь словарь, где ключ — стабильный `section-id` (slot-ключ), значение — единичный writer-объект `{name, repo, section, position}`. Поле `section` несёт заголовок H1 (свободная строка, кириллица допустима), `position` — `top` или `bottom` относительно свободного тела оператора. Заголовок больше не выводится из `group_key.replace("_"," ").capitalize()` в коде. Поддерживается ровно один writer на секцию (audit ловит multi-writer). Layout документа: `[banner] [#status] [top-секции в порядке деклараций] [свободное тело оператора] [bottom-секции] # History`. Layout-правила зафиксированы в `body._restore_owned_and_history` через карту `section_layout` (passed through `compute_inputs` → `TickInputs` → `reapply` → `reassemble`). Umbrella `addenda` удалена.
- **Breaking — формат тега владения**. Тег секции теперь двухкомпонентный: `#expert/<flat-name>/<section-id>` вместо `#expert/<flat-name>`. Уникальная пара `(flat-name, section-id)` идентифицирует секцию в пределах документа. Решает багу «один эксперт в двух секциях молча уничтожает первую при дispatch'е второй». Эксперт никогда не пишет тег сам — диспетчер собирает heading H1 + тег из request-обёртки (`owned_section` + `owned_section_id`) и приписывает к ответному `section_body`. Любой H1, отличающийся от канонического в конфиге, переписывается ассемблером на канонический после reapply.
- **Audit правила под новую схему** — семь новых проверок в `bin/audit.py`: алфавит section-id `^[a-z][a-z0-9_-]*$`; уникальность section-id в пределах класса (через обе umbrella); обязательные поля writer-объекта `name`/`repo`/`section`/`position`; `position ∈ {top, bottom}`; непустой `section` title; алфавит flat-name `^[a-z0-9_-]+$`; разрешимость `name` в корневом каталоге `experts:`. Каждое нарушение — FAIL с указанием пути.
- **`/lazy-review.configure` wizard переписан** под новый question-flow. На каждую добавляемую секцию задаёт пять отдельных вопросов: umbrella (validation | terminal), section-id (валидация против алфавита + уникальность), section title, position (top | bottom), writer-name. Записывает в `experts.<umbrella>.<section-id>` полный writer-объект.
- **Протокол `lazy-review.doc-review-protocol.md` поднят до v4** и ужат до wire-only. Удалены секции про консьюмерский state machine (Layout invariants, Approve gestures, Terminal-state status callout, System callouts vocabulary, Final-as-section lifecycle) — вынесены из протокола в консьюмерскую реализацию. Добавлено: поле `owned_section_id` в request, описание формата двухкомпонентного тега, закрытый enum для `request.role` (`main | history | <section-id> | repair`). Размер сократился с ~183 до ~120 строк.
- **barrier-pipeline для post-approve фазы**. Валидаторы и терминалы теперь проходят барьером, а не каскадом. После одобрения диспетчер ставит в очередь джобы для ВСЕХ валидаторов сразу, ждёт пока каждый отпишется, и за один проход пишет N секционных коммитов (по одному на писателя) + один решающий служебный коммит — вместо прежней логики «первый валидатор с замечанием немедленно откатывает документ», из-за которой второй и третий валидатор за круг не запускались. `review_validation_round` теперь инкрементится раз за полный проход с замечаниями, не раз за валидатора; порог `concerns_decision_threshold` стал осмысленным для multi-validator классов. Та же barrier-модель у терминалов. Фаза барьера хранится в новом frontmatter-ключе `review_phase` (`validators` \| `terminals` \| отсутствует) — явное состояние, а не вывод из commit-сообщений, чтобы операторский ответ на вопрос терминала не переоткрывал закрытый барьер валидаторов. Историк теперь зовётся ровно раз — на approve-mirror коммите (достигнутое чистое одобренное состояние), а не на каждый writer-коммит; finalize гейтится на отсутствие незакрытых historian-job'ов, чтобы поздняя запись не легла в уже закрытый документ. Баннер «Ожидание» обогащён фазой: `Waiting: validators` / `Waiting: terminals` / `Waiting: writer`.
- **Cross-plugin protected sections** — `finalize` now lifts every `#protected/<owner>/…` H1 section out before its body-wide strips and restores it byte-for-byte, so a foreign plugin's region (e.g. the wiki's `# See also` block, inner HTML markers included) survives finalize verbatim instead of being eaten by the `html` marker strip. The abandoned lifted-record archive is removed (`#review/lifted-concerns` rename + finalize strip, `rename_owned_section`, the `Tag.LIFTED_CONCERNS` constant): validator concern sections persist `#expert`-tagged as audit-context until stripped by ownership at the next approve-mirror.
